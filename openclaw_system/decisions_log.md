# Decisions Log

## Решение 1
- Решено фиксировать архитектуру и этапы системы на диске, а не держать их только в диалоге.
- Причина: нужна постоянная операционная конфигурация.

## Решение 2
- Решено строить workflow строго по этапам development_plan.md.
- Причина: избежать хаотичного роста системы и смешивания недоделанных контуров.

## Решение 3
- Решено сделать review обязательным перед завершением задачи.
- Причина: без review нет отдельного слоя контроля качества.

## Решение 4
- Решено выделить structured project memory как отдельный слой в openclaw_system.
- Причина: текущее состояние, прогресс, решения и активный контекст должны храниться в понятной и отдельной структуре.

## Решение 5
- Решено ввести отдельный task queue слой со статусами pending / running / failed / done.
- Причина: выполнение задач должно опираться не только на narrative memory, но и на явную очередь состояний.

## Решение 6
- Решено зафиксировать GitHub и GitHub Actions как обязательный dev-инфраструктурный слой целевой системы.
- Причина: история изменений и стандартизированные проверки должны быть частью архитектуры, а не внешним необязательным дополнением.

## Решение 7
- Решено фиксировать runtime limitations отдельно от логических ошибок задач.
- Причина: отсутствие бинаря в PATH и проблемы среды нельзя смешивать с ошибками реализации.

## Решение 8
- Решено фиксировать integrations layer отдельно от факта активной установки каждого интеграционного компонента.
- Причина: архитектурная роль интеграции и её реальный runtime-статус должны храниться раздельно.

## Решение 9
- Решено оформить guardrails как отдельный системный слой с risk tiers и approval rules.
- Причина: полезность системы не должна превращаться в неуправляемый риск.

## Решение 10
- Решено фиксировать вспомогательные модели отдельно от основного маршрута и отдельно от факта их реальной рабочей активации.
- Причина: allowlist не равен стабильному production-path и не должен вводить в заблуждение.

## Решение 11
- Решено завершить базовый план отдельным слоем сценарных шаблонов.
- Причина: типовые задачи должны стартовать не из хаоса, а из стандартных операционных рамок.
